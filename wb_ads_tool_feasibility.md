# Sellico / рекламный инструмент WB: логика продукта и проверка реализуемости

**Дата проверки:** 27.05.2026  
**Цель документа:** превратить концепцию рекламного инструмента для Wildberries в реалистичное техническое ТЗ: что можно делать через официальный WB API, что требует ручного ввода/импорта/интеграций, а что нельзя закладывать как надежную основу без риска.

---

## 0. Главный вывод

Инструмент для WB-рекламы реально построить **не просто как автобиддер**, а как **центр управления рекламой, товарами, экономикой и задачами менеджеров**.

Ключевой принцип:

> Сервис должен не просто показывать статистику, а каждый день отвечать: где мы теряем деньги, где можно масштабироваться, где проблема не в рекламе, а в карточке, цене, остатках или марже.

По официальному WB API реально закрывается большая часть ядра продукта:

- получение рекламных кампаний;
- создание/запуск/пауза/остановка/удаление кампаний;
- управление ставками;
- получение рекламной статистики;
- работа с бюджетами и историей рекламных списаний;
- работа с поисковыми кластерами и минус-фразами;
- получение данных по карточкам, ценам, остаткам, заказам, продажам, воронке продаж, финансовым отчетам, тарифам и комиссиям;
- построение рекомендаций, скорингов, задач, отчетов, уведомлений и автопилота на своей стороне.

Но есть ограничения:

- себестоимость, налоги, внутренняя маржа и часть управленческой экономики WB не знает — это нужно вводить вручную, загружать Excel/CSV или подключать 1С/МойСклад/ERP;
- точные конкурентные данные, позиции конкурентов, их продажи, чужая экономика и часть кабинетных виджетов не являются надежно доступными через официальный Seller API;
- браузерное расширение нельзя закладывать как основной способ парсинга личного кабинета продавца, потому что официальный API прямо указывает, что интеграция с личным кабинетом без WB API запрещена условиями. Расширение можно использовать безопаснее: как overlay-помощник, импорт пользовательских выгрузок, UI-ассистент и быстрые кнопки, которые вызывают наш backend, а backend уже работает через официальный API.

Итоговая оценка:

**75–85% сильного продукта можно сделать официально через API + собственную логику + ручной ввод экономики.**  
Расширение браузера стоит использовать осторожно и только как вспомогательный слой, а не как “скрейпер кабинета”.

## 1. Каким должен быть продукт

### 1.1. Не “таблица рекламы”, а “пульт управления продажами”

Большинство сервисов по рекламе дают таблицы: расход, клики, CTR, CPC, ДРР. Это полезно, но не решает главную боль менеджера.

Правильная логика Sellico:

```text
1. Найти слив бюджета.
2. Найти товары, которые можно масштабировать.
3. Понять, где проблема в карточке, а не в рекламе.
4. Понять, где проблема в цене, остатках, рейтинге, отзывах или марже.
5. Дать конкретную рекомендацию.
6. Дать кнопку действия.
7. Сохранить историю: что изменили, почему, кто изменил и какой был результат.
```

Главный экран должен быть не “аналитикой ради аналитики”, а очередью решений:

```text
Сегодня нужно сделать:
- остановить 3 источника слива бюджета;
- снизить ставки по 7 слабым кластерам;
- масштабировать 5 прибыльных товаров;
- пополнить бюджет в 2 кампаниях;
- не трогать 4 товара, потому что проблема в карточке;
- отключить рекламу по 2 товарам, потому что остаток почти закончился;
- добавить 6 прибыльных запросов в SEO карточек.
```

## 2. Пользователи и их задачи

### 2.1. Собственник / селлер

Собственнику не нужны сотни метрик. Ему нужны ответы:

- сколько потратили;
- сколько получили заказов;
- какой ДРР;
- какая прибыльность;
- какие товары сливают деньги;
- какие товары стоит масштабировать;
- где реклама идет на товар без остатков;
- где реклама работает, но товар убыточен;
- что сделал менеджер и был ли эффект.

**Главный экран для собственника:**

| Блок | Что показывает |
|---|---|
| Деньги | расход, выручка, заказы, ДРР, ROAS |
| Потери | расход без заказов, слабые кампании, слабые кластеры |
| Рост | товары и запросы, которые можно масштабировать |
| Риски | остатки, бюджеты, превышение ДРР, сбои API |
| Контроль команды | действия менеджеров, рекомендации, просроченные задачи |

### 2.2. Маркетолог

Маркетологу нужен подробный слой:

- кампании;
- ставки;
- бюджеты;
- места размещения;
- поисковые кластеры;
- показы;
- клики;
- CTR;
- CPC/CPM;
- добавления в корзину;
- заказы;
- средняя позиция;
- расход;
- выручка;
- ДРР;
- ROAS;
- минус-фразы;
- история изменений.

### 2.3. Менеджер маркетплейса

Менеджеру важно видеть связь рекламы с карточкой:

- цена;
- скидка;
- остатки;
- рейтинг;
- отзывы;
- фото/контент;
- конверсия карточки;
- добавления в корзину;
- заказы;
- возвраты;
- оборачиваемость.

Если клики есть, но корзин нет — сервис должен говорить:

> Проблема не в ставке. Покупатель кликает, но карточка/цена/оффер не убеждают.

### 2.4. Руководитель агентства / тимлид

Ему нужен мультиаккаунт:

- все магазины;
- все клиенты;
- кто сегодня в риске;
- кто сливает бюджет;
- кто требует внимания;
- где менеджер не обработал рекомендации;
- где автопилот сработал;
- где нужно ручное вмешательство.

## 3. Основная модель данных

Нельзя строить систему вокруг кампании. Кампания — это не бизнес-единица. Бизнес-единица — товар.

Правильная структура:

```text
Магазин
  → Товар / nmID
    → Себестоимость / маржа / цена
    → Остатки
    → Карточка / контент / рейтинг / отзывы
    → Рекламные кампании
      → Места размещения
      → Ставки
      → Поисковые кластеры
      → Показы / клики / корзины / заказы
      → Расход / выручка / ДРР / ROAS / прибыль
```

Главная сущность в интерфейсе: **товар / nmID**.

## 4. Легенда реализуемости

| Статус | Значение |
|---|---|
| ✅ Реализуемо через официальный WB API | Можно строить как промышленную функцию |
| 🟡 Частично через API | Нужны ограничения, ручной ввод, импорт или доп. интеграция |
| 🟠 Через наш backend / внутреннюю логику | WB не нужен, считаем и храним сами |
| 🔴 Не закладывать как надежную API-функцию | Нет официального API или есть высокий юридический/технический риск |
| 🧩 Расширение браузера | Только вспомогательный слой: overlay, импорт выгрузок, быстрые действия через наш API |

### 5.2. Ставки и автобиддер

| Функция | Реализуемость | Как делать | Комментарий |
|---|---:|---|---|
| Получить текущие ставки | ✅ | Promotion API bids | Базовая функция автобиддера |
| Изменить ставки | ✅ | Promotion API bids | Обязательно: лимиты, задержки, журнал изменений |
| Получить минимальные ставки | ✅ | Promotion API minimum bids | Нужны для проверки “ставка ниже входа в аукцион” |
| Получить рекомендованные ставки | 🟡 | Promotion API recommendations | В документации есть ограничения, например по типам кампаний; не строить весь алгоритм только на рекомендациях WB |
| Ставки по карточкам | ✅ | Promotion API bids | Основной уровень управления |
| Ставки по поисковым кластерам | 🟡 | Promotion API normquery bids | По документации есть ограничения по типу ставки/кампании; нужно проверять режим кампании |
| Массовое изменение ставок | ✅/🟡 | Через очереди и батчи | Нужно учитывать rate limits и задержку применения |
| История ставок | 🟠 | Хранить самим | WB не решает задачу “почему мы изменили ставку”; это должна хранить наша система |
| Откат изменения ставок | 🟠 | Хранить old_value/new_value | Реализуемо, если мы записываем каждое действие |

**Вывод:** автобиддер реализуем, но его нельзя строить как “каждые 5 секунд крутим ставки”. Нужна очередь изменений, rate limit, cooldown и защита от неполных данных.

| Точный рейтинг как на карточке | 🟡 | API/публичные данные/расчет | Проверить на пилоте; не строить критичную логику без валидации |
| Количество отзывов | ✅/🟡 | Feedback API / карточка | Может отличаться от публичной карточки из-за фильтров/периодов |
| AI-анализ отзывов для улучшения карточки | 🟠 | Наша логика | Отличная доп.фича |

**Вывод:** отзывы и вопросы можно использовать для диагностики карточки и задач менеджеру.

### 5.10. Конкуренты и рынок

| Функция | Реализуемость | Как делать | Комментарий |
|---|---:|---|---|
| Цены конкурентов | 🔴/🟡 | Нет надежного Seller API | Можно делать отдельный модуль с правовой проверкой, публичными данными или ручным импортом |
| Позиции конкурентов в выдаче | 🔴/🟡 | Нет надежного Seller API | Не закладывать в ядро MVP |
| Продажи конкурентов | 🔴 | Нет официального доступа | Только внешние оценочные сервисы/модели, не факт |
| Топ запросов по нише | 🟡 | Частично через аналитику/поисковые отчеты/внешние источники | Нужна проверка на реальных токенах |
| Сравнение цены товара с рынком | 🟡 | Можно ручным импортом/внешним модулем | Не делать обязательным для MVP |

**Вывод:** конкурентную разведку не нужно ставить в ядро первой версии. Иначе продукт станет зависеть от нестабильных и юридически рискованных источников.

## 6. Что делать через браузерное расширение

### 6.1. Главное ограничение

Расширение браузера не должно быть основным способом получения данных из личного кабинета продавца WB.

Причина: в официальной документации WB API указано, что продавец должен использовать API-токен для интеграции, а интеграция с личным кабинетом без WB API запрещена условиями. Поэтому скрейпинг кабинета, скрытых внутренних запросов, session cookies и автоматизация действий в кабинете — рискованный путь для коммерческого продукта.

### 6.2. Безопасные сценарии расширения

Расширение можно использовать как вспомогательный слой:

| Сценарий | Статус | Как использовать |
|---|---:|---|
| Overlay-помощник в кабинете WB | ✅ | Показывать рекомендации Sellico поверх интерфейса кабинета |
| Быстрые кнопки действий | ✅/🟡 | Кнопка в расширении вызывает Sellico backend, а backend делает действие через официальный WB API |
| Подсказки менеджеру | ✅ | “Эту кампанию лучше не масштабировать”, “проверь остатки”, “есть слив” |
| Импорт пользовательской выгрузки | ✅/🟡 | Пользователь сам скачивает отчет, расширение/сервис помогает загрузить его в Sellico |
| UI-навигация | ✅ | Быстрый переход к нужной кампании, товару, отчету |
| Автозаполнение форм с подтверждением пользователя | 🟡/🔴 | Только после юридической проверки и без обхода защиты |
| Парсинг личного кабинета | 🔴 | Не рекомендовать как основу продукта |
| Использование cookies/session/token из браузера | 🔴 | Не использовать |
| Автоматизация CAPTCHA/MFA/логина | 🔴 | Не использовать |
| Вызов внутренних неофициальных endpoints WB | 🔴 | Не использовать |

### 6.3. Правильная архитектура расширения

```text
Browser Extension
  → показывает подсказки в кабинете
  → отправляет действие в Sellico Backend
  → Sellico Backend проверяет права, лимиты, стратегию
  → Sellico Backend вызывает официальный WB API
  → Sellico сохраняет audit log
  → Extension показывает результат
```

Так расширение становится UX-надстройкой, а не запрещенным способом интеграции.

## 8. Формулы внутри системы

### 8.1. ДРР

```text
ДРР = Расход на рекламу / Выручка × 100%
```

Нужно считать несколько видов ДРР:

| Метрика | Формула | Зачем нужна |
|---|---|---|
| Рекламный ДРР | рекламный расход / рекламная выручка | Оценка рекламы |
| Общий ДРР товара | рекламный расход товара / вся выручка товара | Оценка влияния рекламы на товар |
| ДРР магазина | весь рекламный расход / вся выручка магазина | Уровень собственника |
| Прогнозный ДРР | расход сегодня / прогноз выручки | Контроль дня |
| Маржинальный ДРР | рекламный расход / маржа | Самый бизнесовый показатель |

### 8.2. ROAS

```text
ROAS = Выручка / Расход
```

ROAS без маржи опасен. Один товар может быть прибыльным при ROAS 5, а другой убыточным при ROAS 10.

### 8.3. CPO

```text
CPO = Расход / Количество заказов
```

CPO часто полезнее CPC, потому что менеджеру важна цена заказа, а не цена клика.

### 8.4. Конверсия из клика в заказ

```text
CR = Заказы / Клики × 100%
```

### 8.5. Добавления в корзину из клика

```text
Cart Rate = Добавления в корзину / Клики × 100%
```

### 8.6. Маржа до рекламы

```text
Маржа до рекламы = Цена продажи - Себестоимость - Комиссия WB - Логистика - Налог - Прочие расходы
```

### 8.7. Максимальный безопасный ДРР

```text
Максимальный безопасный ДРР = Маржа до рекламы / Цена продажи × 100%
```

Пример:

```text
Цена товара: 1 500 ₽
Себестоимость: 600 ₽
Комиссия + логистика + налог: 350 ₽
Маржа до рекламы: 550 ₽
Максимальный безопасный ДРР: 550 / 1500 = 36,6%
```

## 9. Дерево решений

### 9.1. Перед любым повышением ставки

Система должна проверять:

| Проверка | Действие |
|---|---|
| Нет остатков | не повышать, остановить масштабирование |
| Остаток низкий | ограничить бюджет/ставку |
| Маржа неизвестна | запретить агрессивный автопилот |
| Товар убыточен | не масштабировать |
| Кампания без бюджета | уведомить / пополнить при разрешении |
| Низкий рейтинг/мало отзывов | мягкий режим продвижения |
| Цена выше рынка | задача менеджеру, не повышать слепо |
| CTR низкий | задача по главному фото/офферу |
| Клики есть, корзин нет | задача по карточке/цене/отзывам |
| Корзины есть, заказов нет | задача по офферу/доставке/доверию |
| Данных мало | не делать резких действий |
| API данные свежие не полностью | отложить автодействие |

## 10. Статусы товаров

Каждый товар должен автоматически получать статус.

| Статус | Условия | Действие |
|---|---|---|
| Масштабировать | ДРР ниже цели, заказы есть, маржа известна, остатки есть | поднять ставки/бюджет |
| Держать | показатели в норме | не трогать или мягко оптимизировать |
| Снизить ставку | ДРР выше цели, но заказы есть | снизить ставку, оставить рабочие кластеры |
| Остановить | расход есть, заказов нет, данных достаточно | снизить/остановить/минусовать |
| Проблема карточки | показы/клики есть, CTR/Cart Rate слабые | задача контенту/менеджеру |
| Проблема цены/оффера | клики/корзины есть, заказов нет | проверить цену, доставку, отзывы |
| Нет остатков | товар рекламируется, но остаток низкий/0 | остановить масштабирование |
| Недостаточно данных | мало показов/кликов/заказов | продолжить тест с лимитом |

## 11. Кластеры: классификация

### 11.1. Winners / победители

Условия:

```text
Есть заказы
ДРР ниже цели
Корзины есть
CTR нормальный
```

Действия:

```text
- повышать ставку;
- держать позицию;
- выносить в приоритет;
- использовать как SEO-запрос;
- не минусовать.
```

### 11.2. Watch / наблюдение

Условия:

```text
Есть клики
Есть корзины
Заказов мало или данных недостаточно
```

Действия:

```text
- не отключать;
- дать добрать статистику;
- ограничить ставку;
- проверить через 1–3 дня.
```

### 11.3. Losers / убыточные

Условия:

```text
Расход есть
Клики есть
Заказов нет или ДРР высокий
```

Действия:

```text
- снизить ставку;
- после порога данных отключить или минусовать;
- создать причину решения.
```

### 11.4. Trash / мусор

Условия:

```text
Запрос нерелевантен товару
Клики есть
Корзин нет
Заказов нет
```

Действия:

```text
- предложить минус-фразу;
- не ждать слишком долго, если запрос явно нерелевантный;
- логировать решение.
```

### 11.5. SEO ideas

Условия:

```text
Кластер дает заказы
ДРР хороший
Запрос релевантный
```

Действия:

```text
- создать задачу добавить запрос в название/описание/характеристики;
- предложить отдельный слайд/визуальный акцент в карточке;
- отметить запрос как коммерчески ценный.
```

## 12. Автобиддер: логика действий

### 12.1. Сценарий: есть показы, нет кликов

```text
Показы высокие
CTR низкий
Кликов мало
```

Вывод:

```text
Проблема в привлекательности карточки, фото, цене или нерелевантном показе.
```

Действия:

```text
- не повышать ставку;
- проверить главное фото;
- проверить цену;
- проверить название;
- проверить рейтинг;
- снизить слабые кластеры;
- создать задачу менеджеру.
```

### 12.2. Сценарий: есть клики, нет корзин

```text
Клики есть
Корзин мало
Заказов нет
```

Вывод:

```text
Покупатель заинтересовался, но карточка/цена/отзывы/комплектация не убеждают.
```

Действия:

```text
- снизить ставку на 10–25%;
- не отключать сразу, если данных мало;
- создать задачу: проверить цену, фото, отзывы, комплектацию, доставку;
- если кликов много и корзин нет — отключить/минусовать кластер.
```

### 12.3. Сценарий: есть корзины, нет заказов

```text
Клики есть
Корзины есть
Заказов нет
```

Вывод:

```text
Проблема ближе к офферу: цена, доставка, остатки, размер, комплектация, доверие.
```

Действия:

```text
- не резать резко;
- проверить цену и доставку;
- проверить наличие вариантов;
- проверить отзывы;
- снизить ставку мягко.
```

### 12.4. Сценарий: есть заказы, ДРР ниже целевого

```text
Заказы есть
ДРР ниже цели
Остатки есть
Маржа позволяет
```

Вывод:

```text
Можно масштабировать.
```

Действия:

```text
- поднять ставку на 5–15%;
- добавить бюджет;
- расширить кластеры;
- удерживать позицию;
- пометить товар как “рост”.
```

### 12.5. Сценарий: есть заказы, но ДРР выше целевого

```text
Заказы есть
ДРР выше цели
Маржа съедается
```

Вывод:

```text
Продажи есть, но трафик дорогой.
```

Действия:

```text
- снизить ставку на 5–20%;
- оставить только кластеры с лучшим CPO;
- отключить убыточные кластеры;
- проверить расход без продаж.
```

### 12.6. Сценарий: нет показов

```text
Показов мало
Бюджет есть
Ставка ниже минимальной/конкурентной
```

Вывод:

```text
Кампания не входит в аукцион или не активна.
```

Действия:

```text
- проверить статус кампании;
- проверить бюджет;
- проверить наличие товара;
- проверить минимальную ставку;
- поднять ставку постепенно, если товар готов к рекламе.
```

## 13. Автопилот: уровни доверия

### Уровень 1. Только аналитика

Сервис ничего не меняет. Только показывает:

```text
- проблемы;
- рекомендации;
- прогноз;
- потенциальную экономию;
- потенциальный рост.
```

### Уровень 2. Полуавтомат

Сервис предлагает действия, менеджер подтверждает:

```text
- снизить ставку на 15%;
- поднять ставку на 10%;
- добавить минус-фразу;
- пополнить бюджет;
- остановить кампанию.
```

### Уровень 3. Автопилот с лимитами

Сервис сам меняет ставки, но только в рамках правил:

```text
- не повышать ставку больше чем на 15% за раз;
- не менять ставку чаще 1 раза в 2 часа;
- не превышать дневной бюджет;
- не поднимать ДРР выше лимита;
- не трогать товары без маржи;
- не трогать товары без остатков;
- не делать действие при неполных данных.
```

### Уровень 4. Полный автопилот

Для зрелых магазинов:

```text
- сам повышает;
- сам снижает;
- сам минусует;
- сам перераспределяет бюджет;
- сам останавливает кампании;
- сам отправляет отчет.
```

Даже в полном автопилоте обязательно:

```text
- журнал изменений;
- возможность отката;
- лимит максимальной ставки;
- лимит дневного расхода;
- уведомления;
- объяснение причины действия.
```

## 14. Защитные правила автопилота

Примеры правил:

```text
Не повышать ставку, если за последние 24 часа меньше 20 кликов.
```

```text
Не отключать кластер, если расход меньше 1,5 целевого CPO.
```

```text
Не повышать ставку больше чем на 15% за одно действие.
```

```text
Не делать больше 3 изменений по одному товару в день.
```

```text
Не масштабировать товар, если маржа неизвестна.
```

```text
Не масштабировать товар, если остаток меньше минимального порога.
```

```text
Не менять ставки, если свежесть данных хуже допустимого окна.
```

```text
Не выполнять массовые действия, если WB API вернул серию ошибок/429.
```

## 15. История изменений

Каждое действие должно сохраняться:

```text
Дата / время
Кто сделал: пользователь или автопилот
Что было
Что стало
Почему изменено
Какая метрика повлияла
Какая была стратегия
Какой был прогноз
Что произошло после изменения
Можно ли откатить
```

Пример:

```text
26.05.2026 11:40
Товар: Петля мебельная 35 мм
Кластер: петля мебельная с доводчиком
Ставка была: 320 ₽
Ставка стала: 360 ₽
Причина: ДРР 7,8% при целевом 15%, средняя позиция 6,4, остатки 430 шт.
Результат через 24 часа: заказы +18%, ДРР 9,1%.
```

Это критично для доверия. Менеджер и собственник должны видеть не только результат, но и причину действий.

```text
→ осторожно, реклама работает, но товар может упереться в проблемы.
```

## 26. Технические риски

| Риск | Что делать |
|---|---|
| Изменения WB API | Версионировать коннекторы, мониторить ошибки, быстро обновлять адаптеры |
| Rate limits / 429 | Scheduler, token bucket, retry/backoff, приоритеты запросов |
| Задержка данных | Показывать freshness, не принимать решения по устаревшим данным |
| Неполная атрибуция рекламы | Разделять рекламную выручку, общую выручку и маржинальный эффект |
| Себестоимость не из WB | Ручной ввод, Excel/CSV, ERP/1С/МойСклад |
| Товарные остатки отличаются по схемам | Отдельная логика FBO/FBS/DBS |
| Скрейпинг кабинета запрещен/рискован | API-first, extension только как UX-слой |
| Автопилот может навредить | Лимиты, audit log, откат, режимы доверия |
| Менеджеры не доверяют системе | Объяснять каждое решение и показывать результат после действия |
