# ADR-0002: HS256 (HMAC-SHA256) JWTs for session tokens

- **Status**: Accepted
- **Date**: 2026-03-28
- **Authors**: backend team
- **Deciders**: tech lead, security

## Context

The API needs short-lived access tokens (15 min) and longer-lived refresh
tokens (168 h). At the time of decision, the only consumer of these tokens
is our own backend (the same process that issues them) — no external
service needs to verify them independently.

## Decision

Use HMAC-SHA256 (HS256) signed JWTs with a single shared secret loaded
from `JWT_SECRET` env (≥ 32 bytes). Refresh tokens are stored in an
HttpOnly cookie set by `/api/v1/auth/login` and `/api/v1/auth/refresh`;
access tokens travel in the `Authorization: Bearer` header.

## Alternatives considered

| Option | Pros | Cons | Why rejected |
|--------|------|------|--------------|
| HS256 + shared secret (chosen) | Simple; fast; one rotation primitive | Symmetric — anyone with the secret can mint tokens | Acceptable: only one issuer (the API itself) |
| RS256 / EdDSA + key pair | Public key can be safely distributed | Adds key management; extra dependency | No third-party verifier needs the public key today |
| Opaque tokens stored in Redis | Trivially revocable; no secrets in token | Every request becomes a Redis round-trip | Auth latency would be significant overhead at scale |
| Sessions in PostgreSQL | Same revocation story; one less moving part | Same per-request DB hit | Worse latency than Redis with no benefits |

## Consequences

- **Good**: zero-overhead verification (HMAC, no DB); access tokens are
  stateless; refresh-rotation makes the access-token blast radius small.
- **Bad**: token revocation requires either a deny-list (we don't keep one)
  or waiting up to 15 min for natural expiry. For "kill this user's
  sessions now" we delete the refresh token from DB so the next refresh
  fails — acceptable for the threat model.
- **Migration path**: if a future external service (e.g. a third-party
  partner) needs to verify tokens, switching to RS256 means rotating
  `JWT_SECRET` to a private key and exposing the public key at JWKS
  endpoint. The token format itself doesn't need changing. Plan ahead with
  ADR-XXXX when that consumer appears.

## How we'll know it worked

- Auth latency p99 < 5 ms (HMAC verify is microseconds)
- No security incident traced to symmetric-key compromise (annual review)
